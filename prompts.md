**Sequential Development Prompts for Freight Bidding Marketplace**

**Phase 1: Project Setup & Database Design**

**Prompt 1: Project Initialization**

Create a new React.js project with Supabase backend integration. Set up the project structure with separate folders for components, pages, services, hooks, and utilities. Configure environment variables for Supabase connection (API URL and anon key). Install necessary dependencies including React Router for navigation, Tailwind CSS for styling, and Supabase client library. Set up a basic routing structure with placeholder pages for authentication, shipper dashboard, carrier dashboard, driver dashboard, broker dashboard, and admin panel.

**Prompt 2: Database Schema Design - Core Tables**

Design a comprehensive Supabase PostgreSQL database schema with the following core tables:

**users table**: Store user account information with fields for id (UUID primary key), email (unique), password hash, phone number, created timestamp, updated timestamp, and account status (active/suspended/deactivated).

**user_roles table**: Enable multi-role functionality with fields for id (UUID primary key), user_id (foreign key to users), role_type (enum: shipper, carrier, driver, broker), is_active (boolean), verification_status (enum: pending, verified, rejected), verification_documents (JSONB), role_specific_profile (JSONB for storing role-specific details), created timestamp, and activated timestamp.

**user_preferences table**: Store user preferences with fields for user_id (foreign key to users, primary key), language (default: 'en'), currency (default: 'XAF'), notification_preferences (JSONB), and timezone.

Include proper indexes on foreign keys, email, and frequently queried fields. Set up Row Level Security (RLS) policies to ensure users can only access their own data. Create triggers for automatic timestamp updates.

**Prompt 3: Database Schema - Shipment & Bidding Tables**

Extend the database schema with shipment and bidding related tables:

**shipments table**: Fields for id (UUID primary key), shipper_user_id (foreign key to users), shipment_number (unique, auto-generated), pickup_location (text), pickup_latitude (decimal), pickup_longitude (decimal), delivery_location (text), delivery_latitude (decimal), delivery_longitude (decimal), scheduled_pickup_date (timestamp), scheduled_delivery_date (timestamp), freight_type (text), weight_kg (decimal), dimensions_json (JSONB for length/width/height), quantity (integer), special_handling_requirements (text), preferred_vehicle_type (text), insurance_required (boolean), insurance_value (decimal), loading_requirements (text), unloading_requirements (text), status (enum: draft, open_for_bidding, bid_awarded, in_transit, delivered, cancelled), created_at, updated_at.

**bids table**: Fields for id (UUID primary key), shipment_id (foreign key to shipments), carrier_user_id (foreign key to users), bid_amount (decimal), estimated_delivery_date (timestamp), bid_breakdown_json (JSONB for cost details), bid_status (enum: active, withdrawn, outbid, awarded, rejected, expired), bid_message (text), bid_submitted_at (timestamp), bid_expires_at (timestamp), auto_bid_enabled (boolean), max_auto_bid_amount (decimal), bid_ranking (integer), is_counter_offer (boolean), original_bid_id (foreign key to bids, nullable).

**bid_history table**: Track all bid activities with fields for id (UUID primary key), bid_id (foreign key to bids), action_type (enum: submitted, modified, withdrawn, outbid, awarded, expired), action_timestamp (timestamp), previous_amount (decimal), new_amount (decimal), action_by_user_id (foreign key to users).

Create appropriate indexes, foreign key constraints, and RLS policies. Set up database triggers to automatically update bid rankings when new bids are submitted.

**Prompt 4: Database Schema - Vehicle & Driver Management**

Create tables for fleet and driver management:

**vehicles table**: Fields for id (UUID primary key), carrier_user_id (foreign key to users), vehicle_type (text), make (text), model (text), year (integer), registration_number (unique), license_plate (unique), capacity_kg (decimal), capacity_cubic_meters (decimal), insurance_policy_number (text), insurance_expiry_date (date), gps_device_id (text), last_maintenance_date (date), next_maintenance_due_date (date), vehicle_status (enum: active, maintenance, inactive), created_at, updated_at.

**driver_assignments table**: Fields for id (UUID primary key), driver_user_id (foreign key to users), vehicle_id (foreign key to vehicles), carrier_user_id (foreign key to users), assignment_start_date (timestamp), assignment_end_date (timestamp, nullable), is_active (boolean).

**shipment_assignments table**: Fields for id (UUID primary key), shipment_id (foreign key to shipments), driver_user_id (foreign key to users), vehicle_id (foreign key to vehicles), assigned_at (timestamp), accepted_at (timestamp, nullable), started_at (timestamp, nullable), completed_at (timestamp, nullable), assignment_status (enum: pending, accepted, rejected, in_progress, completed, cancelled).

**driver_documents table**: Fields for id (UUID primary key), driver_user_id (foreign key to users), document_type (enum: license, national_id, professional_cert, medical_cert), document_number (text), document_url (text for Supabase storage path), issue_date (date), expiry_date (date), verification_status (enum: pending, verified, rejected), uploaded_at (timestamp).

Include indexes on foreign keys and status fields. Set up RLS policies for carrier-driver-vehicle relationships.

**Prompt 5: Database Schema - Payments & Financial Management**

Design the payment and financial tracking tables:

**transactions table**: Fields for id (UUID primary key), transaction_type (enum: shipment_payment, refund_full, refund_partial, commission_settlement), shipment_id (foreign key to shipments, nullable), payer_user_id (foreign key to users), payee_user_id (foreign key to users), gross_amount (decimal), platform_commission_amount (decimal), platform_commission_percentage (decimal), aggregator_fee_amount (decimal), aggregator_fee_percentage (decimal), mobile_money_fee_amount (decimal), mobile_money_fee_percentage (decimal), total_deductions (decimal), net_amount (decimal), currency (text, default 'XAF'), payment_method (enum: orange_money, mtn_momo, card, bank_transfer), aggregator_transaction_id (text, unique), payment_status (enum: pending, processing, completed, failed, refunded_full, refunded_partial), payment_initiated_at (timestamp), payment_completed_at (timestamp), created_at.

**invoices table**: Fields for id (UUID primary key), transaction_id (foreign key to transactions), invoice_number (unique, auto-generated), invoice_type (enum: payment, refund_full, refund_partial), issued_to_user_id (foreign key to users), invoice_data_json (JSONB storing complete breakdown), pdf_url (text for Supabase storage), issued_at (timestamp), sent_at (timestamp, nullable).

**carrier_wallets table**: Track carrier earnings by role with fields for user_id (foreign key to users), role_type (enum matching user_roles), gross_earnings (decimal, default 0), total_commissions_paid (decimal, default 0), total_aggregator_fees_paid (decimal, default 0), total_mobile_money_fees_paid (decimal, default 0), net_earnings (decimal, default 0), pending_amount (decimal, default 0), last_updated (timestamp). Use composite primary key (user_id, role_type).

**refund_requests table**: Fields for id (UUID primary key), transaction_id (foreign key to transactions), shipment_id (foreign key to shipments), requested_by_user_id (foreign key to users), refund_type (enum: full, partial), refund_percentage (decimal, nullable for partial), refund_reason (text), refund_status (enum: pending, approved, rejected, processing, completed), evidence_urls_json (JSONB array of file URLs), admin_notes (text), requested_at (timestamp), reviewed_at (timestamp, nullable), processed_at (timestamp, nullable).

Set up database functions to automatically calculate net amounts and update wallet balances. Create triggers for transaction validation.

**messages table**: Fields for id (UUID primary key), conversation_type (enum: shipment_chat, bid_negotiation, support_ticket), related_shipment_id (foreign key to shipments, nullable), related_bid_id (foreign key to bids, nullable), sender_user_id (foreign key to users), receiver_user_id (foreign key to users), message_content (text), message_type (enum: text, image, document), attachment_url (text, nullable), is_read (boolean, default false), sent_at (timestamp).

**ratings_reviews table**: Fields for id (UUID primary key), shipment_id (foreign key to shipments), reviewer_user_id (foreign key to users), reviewed_user_id (foreign key to users), reviewer_role (enum: shipper, carrier, driver), rating_overall (integer 1-5), rating_timeliness (integer 1-5, nullable), rating_communication (integer 1-5, nullable), rating_condition (integer 1-5, nullable for freight condition), review_text (text, nullable), review_status (enum: pending, published, flagged), created_at (timestamp), published_at (timestamp, nullable).

**notifications table**: Fields for id (UUID primary key), user_id (foreign key to users), for_role_type (enum matching user_roles, nullable to indicate which role this notification is for), notification_type (enum: bid_received, bid_outbid, bid_awarded, payment_received, shipment_update, message_received, document_expiring), title (text), message (text), related_entity_type (text), related_entity_id (UUID), is_read (boolean, default false), sent_via_email (boolean), sent_via_sms (boolean), sent_via_push (boolean), created_at (timestamp).
